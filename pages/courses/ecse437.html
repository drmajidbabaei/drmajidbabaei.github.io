Lab Instructions and Files:

  • Lab 1 (Lab_1.pdf) [due Sep 20th]
    In this lab, there are 12 (small) exercises. If you have never worked with any version control systems like Git the main challenge for you will be to understand the concepts, create an account and worke with the cloned repository. Your main task in this lab is to understand what kinds of benefits you will get by using a version control system rather than keep naming your file like: final.cpp, final2.cpp, reallyFinal.cpp, finalyReallyFinal.cpp, OMGfinal.cpp, etc. It is never ending! You never know which one is the actual final cpp file!

  • Lab 2 (Lab_2.pdf) [due Oct 04th]
    In this lab, there is only three exercises. During this lab, you will be using Gerrit, GitHub and GerritHub to add a reviewer, make a change and ask the reviewer manually/automatically to check your changes. Then you will get a notification and try to address the feedback from the reviewer.
    In one part of this lab you should add TA and professor as the reviewer of your code! make sure you do that because if we don't get the message, you don't get the full mark!

  • Lab 3 (Lab_3.pdf, Lab_3.zip) [due Oct 29th]
    This lab is challenging because it is your first time writing Makefile. In the first part of this lab, we will be working on a simple compilation process with just one source file. We will learn three main elements of a Makefile and how we can use them towards our goal which is (usually) generating an executable file. In the second part of this lab, we will consider a bit more complex scenario, where the source code is broken down into multiple files. It is more challenging because now we should first compile those files to object code, an intermediate compilation stage. Next we should compile multiple object files to assemble an executable file.

Final Project

Students will work on their final project in the same teams that they have worked on labs assignment. The task of each group is to use the concepts and tools that they have learned through the course to design, develop, test, and deploy a software project and provide a demo of their project. The following points need to be addressed in the final presentation of your project.

  • The software has to have at least seven components (or classes) that are coded into different files.
  • To manage different version of the software a version control systems like git must be used
  • You should demonstrate that you code has multiple branches, e.g., main, bugfix/‚Ä¶,feature/‚Ä¶, etc. that are merged into your main line of development every couple of days.
  • One student mush act as a code reviewer and send his/her feedback to the other student using code review system like Gerrit or GerritHub
  • Each team must demonstrate how unitary and integration testing performed to ensure the software is free of bugs in each release.
  • Each team must use an automatic build system.
  • Each team must use a static code analysis approach in their project.
  • Each team must use a continuous integration orchestration tool like Jenkins to build artifacts throughout the pipeline.

Note: You are free to choose any programming languages that you are more familiar with.
